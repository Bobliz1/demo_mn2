# MN-BO 项目 AI 协作规则

在本项目（demo_mn2）的所有开发和维护过程中，AI 助手必须绝对遵守以下操作规范：

1. **状态同步（每日日志）**：对项目做出的任何代码修改、超参数调整或环境变更等，均必须在完成改动的同时，记录到 `states.md` 文件中。以当天的日期（如 `## 2026-05-06`）为标题，按天记录。**新条目必须严格按照时间线顺序向后追加**，确保章节内部逻辑一致，严禁乱序插入。
2. **历史保留（回退规范）**：如果有一些之前的改动被决定回退、作废或重构，**绝对不能**在 `states.md` 或其他记忆文档中直接删除原有的记录。必须使用 Markdown 的删除线（例如：`~~被废弃的改动~~`）将原本的文本划掉。这样做是为了保留完整的迭代历史和“踩坑”记录。
3. **推送前置同步（README 更新）**：在每一次执行 `git push` 操作之前，AI 助手必须阅读 `states.md` 文档以获取最新的项目状态，并据此同步更新 `README.md` 文档，确保项目的对外展示说明与内部开发日志保持高度一致。
4. **平衡且专业的沟通**：AI 助手应保持专业、客观且清晰的沟通语气。既要避免夸张的吹捧，也要避免过于书面化或学术化的陈述。沟通风格应类似于科研小组的内部讨论：直接、高效、注重技术逻辑，避免使用过于口语化或不规范的表述。
5. **代码修改前置确认**：在执行任何代码修改（Edit/Replace）之前，AI 助手必须首先列出本次修改计划涵盖的**任务清单**（例如：1. 修改预算逻辑；2. 添加早停模块）。在用户明确指示“执行全部”或“仅执行某几项”之后，AI 助手方可开始编写代码。
6. **实验授权原则**：你可以根据需求主动修改 `.py` 代码或脚本逻辑，但**严禁擅自启动命令行实验**。所有命令行运行操作（尤其是大规模扫参）必须在得到用户显式的“OK”或相关执行指令后方可进行。
7. **熔断与重启规范**：如果命令行实验因 Bug 导致中断，你可以修复代码 Bug，但**禁止修复后自行重新跑实验**。必须向用户汇报 Bug 情况与修复方案，并在再次得到用户“OK”授权后，方可重新启动实验。
8. **README 累积更新规范**：在同步更新 `README.md` 时，必须保留以往版本的核心改进内容和里程碑记录。严禁在更新到新版本时将旧版本的架构说明或重要结论物理删除。应采用“版本演进”或“历史归档”章节的形式，确保 `README.md` 能够完整展示项目从 v1.0 到当前版本的技术进化轨迹。
